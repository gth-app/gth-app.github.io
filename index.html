Creating a Repository with GitHub Desktop

A practical mini-course that walks you through creating a repository, committing changes, working with branches, and synchronizing with GitHub — all without touching the command line.

Tip

Keep this page open while you work in GitHub Desktop. The steps are written so you can follow them in order without switching between tools too often.

Step 1. Prepare a sample CSV file

First, you’ll create a small data file that will be tracked by Git. A simple CSV file is perfect for this.

1
Create a small table
Any spreadsheet editor will do
  1. Create a table in Excel, LibreOffice Calc, Google Sheets, or another editor.
  2. Fill it with dummy data: numbers, dates, categories — anything that looks like a small dataset.
  3. Keep it small and readable (a few dozen rows are enough).
2
Export as CSV
GitHub renders CSV nicely
  1. Export or save the table as a CSV file.
  2. Choose comma as the separator if the editor asks.
  3. Name it something like sample-data.csv.
Check the file format: the extension should be .csv, not .xlsx or another native spreadsheet format. This allows GitHub to preview the table directly in the browser.

Step 2. Create a local repository in GitHub Desktop

Next, you’ll create a new Git repository on your computer using the GitHub Desktop interface.

3
New repository on your hard drive
No terminal required
  1. Open GitHub Desktop.
  2. In the menu, choose “Create a New Repository on your Hard Drive…”.
  3. Fill in:
    • a repository name (for example, csv-demo);
    • a local folder where the repository will live;
    • an optional description if you like.
  4. Confirm the creation of the repository.

At this point, the repository exists only locally on your computer. It is not yet connected to GitHub.

Step 3. Add the CSV file and make your first commit

Now you’ll add your data file to the repository and record the change in Git using a commit.

4
Track your data with Git
From untracked file to committed change
  1. Move or copy your CSV file into the repository folder you just created.
  2. Switch back to GitHub Desktop — it should now show new changes (the file will appear as an untracked or modified file).
  3. In the Summary field, type a message such as “Add sample CSV data”.
  4. Click the Commit button to create your first commit.

The current state of your project is now stored in Git history as a commit and can be referenced or reverted later.

Step 4. Publish the repository to GitHub

Time to put the repository online so that you (and optionally others) can access it from any device.

5
Create the remote repository
Your local repo goes to GitHub
  1. In GitHub Desktop, open the History tab and confirm that your commit is listed.
  2. Click the Publish repository button.
  3. Choose:
    • visibility: Public or Private;
    • a repository name on GitHub (change it if a repo with that name already exists).
  4. Confirm the publication. GitHub Desktop will create a remote repository and push your commits.
  5. When it finishes, use the link in GitHub Desktop to open the repository in your browser.

Step 5. Create a branch, edit the file, and share the branch

Branches let you work on changes safely, without disturbing the main version of your project.

6
Experiment in a feature branch
Keep the main branch clean
  1. Edit your CSV file locally: add a few extra rows or modify existing data.
  2. Look at the diff in GitHub Desktop to see how the file has changed line-by-line.
  3. Create a new branch (for example, feature/update-data).
  4. When prompted, choose the option to bring your current changes into the new branch (often worded as “Bring my changes to the new branch”).
  5. Create a commit in this new branch with a message describing what changed.
  6. Publish the branch to GitHub so others can see and review your changes.
Why branches matter:
Branches let you experiment and develop features separately from the main branch. Once the changes are tested and reviewed, they can be merged back into the main branch.

Step 6. Getting updates from GitHub: fetch and pull

Imagine that you (or a teammate) edited the CSV file directly on GitHub, using the web interface. Your local repository now needs to catch up.

7
See remote changes with fetch
Check what’s new before applying it
  1. Open the repository on github.com and edit the CSV file in the browser.
  2. Commit the change to the master or main branch.
  3. In GitHub Desktop, switch to the same branch (master/main).
  4. Click Fetch origin to check for new commits on the remote.
  5. Compare history on GitHub and in GitHub Desktop: after a fetch, GitHub Desktop knows about remote commits but your local files are not changed yet.
8
Apply the changes with pull
Keep your local copy in sync
  1. Click Pull origin in GitHub Desktop.
  2. GitHub Desktop will download the new commits and merge them into your current branch, updating your local files.
  3. Open the History tab in GitHub Desktop to confirm that the new commits are now part of your local history.
Fetch vs pull:
Fetch only retrieves information about new commits from the server; it doesn’t change your working files.
Pull = fetch + apply the changes to your current branch (by merging or using another update strategy).

Summary

With GitHub Desktop, you can:

  • create local repositories and link them to GitHub;
  • add files and record changes as commits;
  • create and publish branches for safe experimentation and collaboration;
  • push your work to GitHub and stay in sync using fetch and pull.

As you get more comfortable with Git, you can perform the same actions in the terminal with commands like git add sample-data.csv and git commit. Still, the graphical interface remains a powerful and convenient way to understand and visualize what’s happening, especially when you’re just getting started.
